# NADITOS Foundational Gap Analysis — Pass 1 (18 areas)

> Status snapshot of the NADITOS codebase against 18 foundational capabilities
> expected from a mature national transport governance backbone.
> Generated by codebase walk on commit `1d98b0e` (main, post Go 1.25 bump).
> Pass 2 (sovereign-grade axes A–R + cross-cutting S–T) is tracked separately.

## Executive Summary

NADITOS is a **Phase-1 MVP foundation** for a multi-tenant, multi-country national
transport governance platform. The hard architectural work is complete: all 10
microservices exist, the multi-tenant Postgres schema with RLS is fully wired,
JWT + RBAC middleware is production-grade, and the domain model supports vehicle
registry, driver licensing, fines, insurance, inspection, ANPR, and audit. Most
backend services are substantive implementations, but **frontends are largely
skeletal**, and **several modules are worker-only with no HTTP API endpoints**.
The platform runs end-to-end locally and on Fly.io, with integration tests
covering the critical paths.

## 1. Identity & Access Governance — EXISTS (substantive)

**Files:** `packages/go-common/auth/`, `services/auth/`, `services/gateway/`,
`db/migrations/0001_init.up.sql`.

JWT issuance + constant-time HMAC-SHA256 verification (`packages/go-common/auth/jwt.go`).
Claims: subject, tenant_id, role, permissions, device_id, expiry. Bcrypt on
`users.password_hash`; SHA-256 on `refresh_tokens.token_hash`. RBAC via
permission strings (`"fines:create"`) in `role_permissions`; middleware exports
`RequirePermission(perm)` and `RequireAnyRole(roles...)`. Tenant isolation
enforced via Postgres RLS (`SET LOCAL app.tenant_id`) on all tables; auth
middleware verifies tenant header matches JWT claim. Session/device tracking
via `refresh_tokens(device_id, revoked_at)`.

**Gaps:**
- MFA scaffold (`users.mfa_secret`) but **no enrollment/verification endpoints**.
- No explicit live-token session expiry (only refresh expiry).
- No mTLS / API-key service-to-service auth (only JWT).
- No password complexity rules in code.
- **Login audit not wired** — `audit_events` table exists, but
  `auth.handleLogin()` does not call `audit.Emit()`. Login failures/successes
  not logged.
- API keys: MISSING — only JWT bearer.
- Gateway service has no observable routing logic.

**Inconsistency:** registry/fines/license/audit use `packages/go-common/auth`;
insurance/inspection/anpr-gateway have their own auth wiring.

## 2. National Vehicle Registry — EXISTS (substantive)

**Files:** `services/registry/`, `db/migrations/0001_init.up.sql:131-173`,
`db/migrations/0009_vehicle_transfers.up.sql`, `apps/web-admin/app/(app)/vehicles/`,
`apps/web-citizen/app/(app)/vehicles/`.

Vehicle CRUD fully implemented with per-tenant plate-regex validation. VIN,
chassis_number, make, model, year, category, emission_class stored. Ownership
via `owners` (citizen_id or external entity). Stolen-vehicle flag, seized,
wanted booleans. 7-day code-based ownership transfer fully implemented with
outbox events. Status view `v_vehicle_status` computes `green|yellow|red|black`
from insurance/inspection/registration expiry + alerts.

**Gaps:**
- Document uploads framework exists (`fine_evidence`) but no
  `/v1/vehicles/{id}/documents` API at the vehicle level.
- Expiry fields (`insurance_expires_at`, `inspection_expires_at`, etc.)
  writable only from workers (bypassing RLS), not via registry HTTP API.
  Asymmetric write permissions.
- No bulk import (CSV).

## 3. Driver Licensing — EXISTS (substantive)

**Files:** `services/license/`, `db/migrations/0001_init.up.sql:176-194`,
`db/migrations/0003_phase2.up.sql:12-87`.

Driver profiles with `license_number` unique per tenant, classes TEXT[] array.
Endorsements, violations, suspensions tables. Penalty points: `driver_demerit_events`
append-only ledger, `driver_demerit_policy` per-tenant (default 12 points, 24mo
window, 6mo suspension, 36mo reset). Biometric template hash table
(`driver_biometrics`).

**Gaps:**
- Biometric enrollment/verification endpoints: MISSING (table is a placeholder).
- Digital license (QR/NFC): MISSING.
- Testing centers: MISSING (no scheduling or results).
- Suspension enforcement at service level: weak — DB table exists, but no
  consistent runtime check on license lookup.
- License state machine: not enforced at DB level (POST creates a row; no
  pending/active/suspended/revoked transitions).

## 4. Police Enforcement — EXISTS (substantive)

**Files:** `services/fines/`, `services/anpr-gateway/`, `apps/police-pwa/`,
`db/migrations/0001_init.up.sql:225-299`, `db/migrations/0003_phase2.up.sql:89-117`.

Citation issuance: `POST /v1/fines` with vehicle_id, offence_code, evidence
(S3 key), geo, officer_id (from JWT), device_id (from JWT). Server computes
amount from `regulation_offences` (officer cannot override). Duplicate
protection within `duplicate_window_min`. Evidence requirement enforced
(at least one photo). ANPR async pipeline. Police-pwa caches last 14 days of
plate status in encrypted IndexedDB (AES-GCM); fine drafts queued locally.

**Gaps:**
- Roadside check-without-citation: not modeled.
- Patrol activity / shift tracking: MISSING.
- Seizure/impound: only a flag — no warrant, hold duration, release condition
  tracking.
- Offline-mode UI state indicator: not explicit (drafts queue silently).
- ANPR events not flowed through `audit.Emit()` — only `event_outbox`.
  Inconsistent with fines path.

## 5. Fines & Penalty Engine — EXISTS (substantive)

**Files:** `services/fines/`, `services/fines/internal/escalation/engine.go`,
`db/migrations/0001_init.up.sql:109-128`, `db/migrations/0003_phase2.up.sql`.

Per-tenant offense catalog with localized name (JSONB). Escalation: 5-stage
default (+7d warning, +14d penalty, +30d flag, +60d seize, +90d court);
idempotent worker every 5 min. Disputes table with `pending|accepted|rejected|
court`. Payments table with multiple rows per fine.

**Gaps:**
- Installment plans: MISSING (no payment schedule or partial-payment
  tracking).
- Repeat-offender rules at fine level: MISSING.
- Automated appeal rules (appeal window, types): MISSING.
- Anomaly detection job vs. escalation: not cross-linked (a single officer
  fine-farming pattern won't auto-flag during escalation).
- Cancellation requires no documented reason (notes optional).

## 8. Audit & Compliance — EXISTS (substantive)

## Cross-Cutting Findings

### Middleware consistency
Most Go services use `packages/go-common/auth` + `observability` +
`server.Mount()`. Gateway, inspection, insurance, anpr-gateway need a
re-audit for divergent wiring.

### Database tables
~40 tables across 3 migration phases. All user-data tables have `tenant_id`
+ RLS `tenant_isolation`. `naditos_admin` role bypasses RLS for migrations
and workers that write across tenants.

### Tenant model
Multi-tenant via Postgres RLS. Not schema-per-tenant. Phase-6 future
upgrade path noted.

### Error format
`packages/go-common/httpx` provides `WriteErr` / `Err(status, code, message)`.
Pattern is `{"code":"...","message":"..."}`. Not fully audited.

### Integration tests
`go test ./... -race -count=1` per service against a postgres fixture.
**No cross-service integration tests** — each service tested in isolation.

### Smoke test
14-stage end-to-end script (`scripts/smoke.sh`). Currently
`continue-on-error=true` due to "service logs empty" diagnostic issue.

## Most Urgent Gaps (priority-ordered)

1. **Postgres auto-stop on Fly** — first request takes 20–40s; intermittent
   health-check fails. Fix: managed PG or `min_machines_running ≥ 1`.
2. **Branch protection not enforced** — regressions can ship.
3. **Smoke test failing & advisory** — investigate empty-log root cause;
   make required.
4. **Frontend shells lack real screens** — 5/6 apps scaffolded; ministry
   cannot perform workflows.
5. **No real provider adapters** — payments, ANPR, insurance, inspection,
   notifications all DevStub.
6. **No multi-jurisdiction support within a tenant** — needed before any
   real-country rollout where law enforcement is municipal.
7. **Login audit not implemented** — brute-force / unauthorized access
   undetected.
8. **No observability stack deployed** — services emit but nothing collects.
9. **No rate limiting or API throttling** — DoS exposure.
10. **No staging environment** — untested deploys hit prod.
